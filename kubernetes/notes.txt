To use Kubeflow Trainer v2.2 for your JAX SPMD pipeline, you primarily interact with two resources: the TrainJob (your specific run) and the ClusterTrainingRuntime (the template).

-----------------------
from kubeflow.trainer import TrainerClient, CustomTrainer
from kfp import dsl

@dsl.pipeline(name="jax-multi-phase-pipeline")
def jax_pipeline(container_image: str):
    
    # 1. TUNING PHASE
    tune_job = CustomTrainer(
        image=container_image,
        args=["--phase", "tune"],
        runtime="jax-distributed",  # Native v2.2 JAX support
        num_nodes=2,                # SPMD scaling
        resources={"gpu": 1}
    )
    
    # 2. TRAINING PHASE (Runs after Tuning)
    train_job = CustomTrainer(
        image=container_image,
        args=["--phase", "train"],
        runtime="jax-distributed",
        num_nodes=4,                # Scale up for full training
        resources={"gpu": 2}
    ).after(tune_job)
    
    # 3. TESTING PHASE (Runs after Training)
    test_job = CustomTrainer(
        image=container_image,
        args=["--phase", "test"],
        runtime="jax-distributed",
        num_nodes=1,
        resources={"gpu": 1}
    ).after(train_job)

real world cases architectures:
  - remember that a cluster can have 1 or more nodes.
    - a node can have one or more pods
      - a pod can have 1 or more containers but usually
        only 1 to compartimentalize the serice of interest.

Kaggle, behind the scenes:
  - GPUs:
    - a shared node
      - Single-Pod
        - Multi-Container (Sidecar) or Single-Container 
    - the shared node is a high performance GCE 
      bare metal of VM equipped with accelerators like GPUs
      (e.g. Nvidia T4 or P100)
      - a single pod is provisioned for a user's notebook session
        - single GPU container:
          the underlying container runtime maps 1 GPU to the 
          user's container runtime
          (e.g. 1x T4)
        - a multiple GPU container:
          the underlying container runtime maps more than 1 GPUs to the 
          user's container runtime
          (e.g. 2x T4)
  - TPUs:
    (TPU v3-8/vm architectures, or TPU v4)
    - v3-8 means a TPU v3 device with 8 Tensor Cores
      spread across 4 underlying chips.
      - these are handled via Remote Host Service Pattern
        rather than direct pass through container mounts
    - on Kaggle:
      - 2 pods are created: (1) User Pod (the client),
        and (2) TPU node (the server/worker)
        - (1) User Pod:
          - is essentially a CPU only container running
            on Jupyter/Kaggle notebook frontend.
            - the Kaggle environment injects environment
              variables into this container like TPU_NAME
              or a gRPC network address.
            - this local container converts the running
              code's model graph into XLA HLO (high level
              optimizer) bytecode and streams that over
              high speed internet network gRPC to the
              remote TPU node.
        - (2) TPU Node:
          - a dedicated network node managed by GCU TPU
            service.  node runs specialized lighweight
            OS optimized for XLA commands from client.
            - processes the data sent to it and streams
              it back.
