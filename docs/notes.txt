goal is to make a ranker for the recommender systems
and MLOps pipelines for it.

In choosing data loading framework:
(1) Google's grain from the Jax AI Stack.

In choosing the LTR model, there are 3 basic approaches:
(1) gradient boosting
    - good for tabular data, very fast, very accurate
    - cannot handle OOV, that is, cold starts
    - lightGBM, use objective lambdarank
    - xgboost, use objective rank:ndcg or rank:pairwise
    - catboost, use loss function yetirank or pairlogit
    - lightgbm and xgboost have Dask integrations for distributed training
    - MMLSpark for lightgbm for distr training
    - ray for distributed training
(2) DNN listwise ranker
  - TF-Ranking is deprecated
  - Rax (JAx based ranking) is active
(3) graph based listewise ranker
    Graph Transformer (GAT) as graph contextualized listwise ranker:
  - Jraphx + Flax NNX + Rax
    - first:
      GAT2 GNN Encoder (JraphX): 
      Processes the local graph (user + candidates) to capture 
      structural relationships (e.g., "This movie is similar to 3 others you liked").
    - then:
      Listwise Interaction (Flax NNX): 
      A small Transformer block that "compares" all candidates 
      in the list simultaneously.
    - then Ranking Loss (Rax): 
      Optimizes the entire list order using a listwise objective 
      like rax.softmax_loss.
    - handles OOV, is fully inductive

Test, model evaluation Metrics:
- listwise ranking metrics
  NDCG@k, precision@k, recall@k, f1sore@k, MRR@k

In choosing an MLOps pipeline:
- needs to be compatible with Jax AI Stack, cloud compatible
  and agnostic where possible.
- needs to be able to do data parallel distributed training.
- Jax AI stack has fixed protobuf==7.34.1
- KFP requires protobuf>=6.31.1,<7.0
- most common options that work in local and cloud environments:
  - KFP as the orchestrator running in its own virtual env
    + docker container images holding the code which needs
      the JAX AI stack libraries
  - KubeRay as a kubernetes operator or ManagedRay
    + RayTrain, RayTune 
    + docker container images holding the code which needs
      the JAX AI stack libraries
    Ray handles JAX TPU/GPU memory management best.

Monitoring:
- system performance monitoring:
  - Xprof tensorboard profiler
    - visibilitiy into XLA computation graphs
    - high level optimizer operations
    - memory usage on TPUs and GPUs
  - Perfetto
  - prometheus and Grafana
- ML observability and Drift:
  - Arize Phoenix 
    clusters the embeddings using UMAP or HDBScan 
    to look for drift
    - jax compatible
  - DeepEval or Confident AI
    - runs online evaluations to make sure that a common
      failure like recommending the same 5 movies to all
      users does not occur
  - Alibi Detect
    - online drift detection of attention weights

GAT metrics to monitor outside of the listwise ranking metrics:
- attention entropy
  uniform distribution is not informative, but a dirac delta
  function is too focused
- edge density drift
  - monitoring sudden changes in user to movie interactions
- in-degree distributions
  - monitoring for GAT unable to recommend tail features due
    to a few movies dominating the mode
- node homophily
  - whether similar users are still connecting to similar movies

Note: Gemini was used in searching for ranking architectures
and getting to the graph solution.  Then it was most useful
in providing a good start to efficient implementations of
vectorized ops. For example, how to make efficient masks
that operate on a batch with numpy and alternative ways to
make branchless code.  Useful for all aspects of
quick documentaton answers.

-------
in training model:
- if have memory problems:
  - node count is high:  consider neighbor sampling or cluster-GCN
  - edge count is high: consider edge dropout or subgraph sampling
  - feature count is obtrusively high: consider dim reduction in
                                       pre-processing stage or
                                       bottleneck layers
  - multihead attention taking up large amount of memory:
    consider reducing the number of heads or using checkpointing
    with jax.checkpoint
