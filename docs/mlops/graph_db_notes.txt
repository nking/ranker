### The High-Performance Dgraph Query

When a C++ client needs to rank movies for a specific user, 
it runs a single-hop query to fetch the user's historical 
feature arrays along with the target candidates.

```graphql
query getGraphInputs($userUid: string, $candidateIds: [int]) {
  # 1. Fetch User History (Enrichment Step)
  userHistory(func: uid($userUid)) {
    has_history {
      ids: movie_id
      type: movie_type
    }
  }
  
  # Fetch Candidate Features (Target Step)
  candidates(func: eq(movie_id, $candidateIds)) {
    ids: movie_id
    type: movie_type
  }
}

```

C++ Preprocessing Pipeline (`SparseLocalSubgraphTransform`)

Once the C++ client receives the JSON response from Dgraph, 
it avoids graph-traversal algorithms entirely. 
It stacks the history items and target candidates together 
into a single, flat, continuous array chunk.

#### Step A: Reconstruct the Arrays

1. Read the historical `ids` and `type` from the JSON payload into an array.
2. Append the candidate movies directly behind them.
3. Generate the tracking masks: elements from the history array receive 
   a `candidate_mask = false`, while elements from the candidate list 
   receive `candidate_mask = true`.

#### Step B: Build the Topology Natively

Because this is a global user-to-item recommendation graph, your 
edge connection scheme (the sender and receiver arrays) can be hardcoded algorithmically:

* **Senders:** Simply create an array sequence mapping `[0, 1, 2, ... N-1]` representing connections moving outward from the primary user node index.
* **Receivers:** Explicitly direct those indices back to the index tracking the user.

#### Step C: Fixed-Size Array Padding

To keep the TensorFlow SavedModel's XLA engine from crashing or re-compiling, use standard library primitives to pad the structural data right up to your hard allocation boundaries (`MAX_NODES`, `MAX_EDGES`):

```cpp
#include <vector>
#include <algorithm>
#include <tensorflow/core/public/session.h>

// Your locked production signatures
const int MAX_NODES = 128;
const int MAX_EDGES = 512;

void prepare_tf_inputs(const std::vector<int>& raw_ids, 
                        const std::vector<bool>& raw_masks,
                        tensorflow::Tensor& node_ids_tensor,
                        tensorflow::Tensor& node_mask_tensor) {
    
    // 1. Initialize clean, zero-padded C++ vector structures
    std::vector<int32_t> padded_ids(MAX_NODES, 0);
    std::vector<bool> padded_masks(MAX_NODES, false);
    
    // 2. Safely cut or copy your graph components up to budget constraints
    size_t nodes_to_copy = std::min(raw_ids.size(), static_cast<size_t>(MAX_NODES));
    std::copy(raw_ids.begin(), raw_ids.begin() + nodes_to_copy, padded_ids.begin());
    std::copy(raw_masks.begin(), raw_masks.begin() + nodes_to_copy, padded_masks.begin());

    // 3. Map memory space natively to your active TF Input Tensors
    auto tensor_ids_matrix = node_ids_tensor.flat<int32_t>();
    auto tensor_mask_matrix = node_mask_tensor.flat<bool>();

    for (int i = 0; i < MAX_NODES; ++i) {
        tensor_ids_matrix(i) = padded_ids[i];
        tensor_mask_matrix(i) = padded_masks[i];
    }
}

```

### Architectural Performance Checklist

1. **Skip the JSON Parser Overheads:** If your throughput targets are extreme, switch Dgraph's output formatting mode from standard JSON strings to **Protocol Buffers (gRPC)**. The Dgraph C++ Client supports direct gRPC streams, allowing your preprocessing loop to serialize bytes directly into your `std::vectors` without dynamic memory reallocations or parsing text maps.
2. **Maintain Global ID Mappings:** Keep a lightweight, synchronized dictionary or embedding map of `movie_uid -> model_integer_id` loaded inside your C++ client memory. If Dgraph passes back string UIDs (like `0x42`), translating them to integers directly via an in-memory hash map avoids database round-trips.
3. **Static Edge Allocations:** Because the local graph topology for a single user ranking recommendations follows a strict star-topology (User $\leftrightarrow$ Item nodes), you can pre-allocate your `receivers` and `senders` arrays *once* when the C++ client initializes. You only need to dynamically swap out the integer feature attributes during active requests.
