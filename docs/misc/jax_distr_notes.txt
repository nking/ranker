in jax distr:
- a Host is a CPU (Node/Machine) that can manage
  several local devices (accelerators such as TPU or GPU chips)
- a Process is a jax program running on a host.
  in the SPMD grain model (using grain.sharding.ShardByJaxProcess()),
  each process id gets 1/(num_processes) of the data
- Local Devices: These are the specific accelerators (e.g., GPU 0 and GPU 1)
  that a single JAX process can directly control without needing 
  to communicate over the network.  
- Global Devices: The collection of all devices available across the 
  entire distributed cluster (all processes and all hosts).  
- backend:
  the specific hardware for which jax compiles code to run on.
  JAX backends: CPU, GPU, TPU
- physical device:
  - silicon hardware attached to the server
- Compute Unit (CU)
  - on a CPU these are physical cores
  - on a GPU these are streaming multiprocessors (SM)
  - on a TPU core
    - there is 1 or more Matrix Multiply Units (MXUs) for fast dense matrix multiplications
    - there is 1 Vector Processing Unit (VPU) for vector ops that don't fit in MXU, like activations,
      pooling, and normalization
    - there is 1 Scalar Unit for the control flow, memory addressing, and basic scalar
      arithmetic necessary to keep the MXU and VPU fed with data.
- Logical Device
  - the software abstraction to distribute computation
  - jax.devices() lists the logical devices
  - jax usually has 1 logical device per accelerator.
    e.g. 8 phyiscal GPUs is 8 logical devices
  - pmap and sharding targets logical devices
- Local Device
  - a logical device that is attached to a host (node) which
    is running a process.
- Global Device:
  - any logical device in the cluster.
  - to contact the memory of another global device, jax
    coordinates it over the network, via XLA.
  - jax.devices() lists all global devices

why multi-host sharding synchronization is needed:
- a TPU Pod or a distributed GKE setup need multi-host sharding synchronization
before oprations such as checkpoint saves or restores because each worker
may have made different progress through their respective lazily instantiated
model graphs, for example, hence have different structures that have been instantiated.  
The process of saving a checkpoint needs all workers to agree on the same
overall structure for what is being saved, so synchronization needs to occur
before orbax can write.
- to facilitate elastic training, where for example, a model is training on 
a 16-node TPU cluster that becomes unavailable, and training is resumed on
a 32-node TPU cluster.  the global_model_target and mesh implemented in train.py
handles the rerouting of the 16 original shards into the 32 current shards.
- note that orbax under the hood uses a coordinator-worker architecture.
only host 0 writes the metadata and config files and workers write their
specific byte-ranges of the sharded tensor arrays asynchronously.
- note that the optimizer might be taking 2-3 times more space than the
model weights and so the same synchronization is required for them.
